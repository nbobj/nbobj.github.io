  4. NamesrvController#initialize

    从方法就知道，这是一个初始化NamesrvController对象的方法。到底初始化什么，看源码：

    在这方法中主要做了：

    1. 创建NettyRemotingServer对象，这个对象是对ServerBootstrap的进一步封装。这个类会接受两个参数，一个是NamesrvConfig另一个是ChannelEventListener，其中第二个参数在NameServer中对应的实现类就是BrokerHousekeepingService，这个类的作用就是在start中说

    2. 创建了一个remotingExecutor现场池，这个线程池用来处理producer和Consumer的具体请求。

      producer和Consumer的网络事件仍然是交给netty的pipeline处理的，只是在这个pipeline的最后的一个channelHandler会把处理的流程封装成一个Runnable，然后交个这个线程池处理，这是一个典型的多Multiple Reactor模式，将线程封为IO线程和业务线程，将线程的职责分开。

    3. 接着到NamesrvController.registerProcessor方法，就是为NettyRemotingServer添加一个默认的请求处理器defaultRequestProcessor。

    4. 添加了两个定时任务，其中最重要的的一个定时任务

      用来每隔10s扫描一次Broker,移除不活跃的Broker

    5. 创建了一个FileWatchService对象，这个对象实际上就是一个线程对象，这个线程的逻辑是监听

      这三个文件是否变化了，如果变化了，就新建一个SslContext对象，而这个对象是netty中提供的，是用来创建SslHandler，而这个Handler就是ChannelHandler，是用来对请求做解密，对响应做加密的。

  5. Runtime.getRuntime().addShutdownHook

    jvm中增加一个关闭的钩子，当jvm关闭的时候，会执行系统中已经设置的所有通过方法addShutdownHook添加的钩子，当系统执行完这些钩子后，jvm才会关闭。所以这些钩子可以在jvm关闭的时候进行内存清理、对象销毁等操作。

    这种模式在很多的框架源码中使用

    总结

    可以看到，NameServer的启动主要就是

    1. 创建两个配置类

      • NamesrvConfig
      • NettyServerConfig

    2. 用这个1中的两个配置类，创建了NamesrvController。

    3. 执行NamesrvController的initialize方法，来初始化NettyRemotingServer，并且为这个remotingServer指定了NameServer自己的NettyRequestProcessor，并为这个处理器分配一个线程

    4. 接着启动了一个每隔10s扫描一次Broker,移除不活跃的Broker的定时任务

    5. 最后，通过NettyRemotingServer的start方法，执行了ServerBootstrap的bind方法，然后启动NettyEventExecutor线程，最后启动了一个每秒处理响应的定时任务。

响应

Topic路由注册、剔除机制

在NameServer中会维护这些集合

  • topicQueueTable：Topic消息队列路由信息，
  • brokerAddrTable：Broker基础信息
  • clusterAddrTable：Broker集群信息
  • brokerLiveTable：Broker状态信息
  • filterServerTable：Broker上的FilterServer列表

在NameServer和Broker都启动后，Broker向NameServer以每30s的频率向NameServer发送心跳包。在NameServer中通过看下图的DefaultRequestProcessor代码快处理这个心跳包。

通过这个方法后，上边的集合中就保存了Broker的信息。

而且NameServer也会在启动时，开启一个定时任务以每10s的频率扫描Broker注册表，移除在连续120s内未收到的Broker。

注：NameServer的实现基于内存，NameServer并不会持久化路由信息，持久化的重任是交给Broker来完成。
